1. Giới thiệu dự án

LiemResearch là một nền tảng hỗ trợ sinh viên, giảng viên và nhà nghiên cứu tra cứu bài báo khoa học theo ngữ nghĩa, phân tích xu hướng chủ đề và tự động tổng hợp báo cáo chuyên sâu. Mọi kết luận từ AI đều được đối chiếu với cơ sở dữ liệu học thuật thực tế để đảm bảo tính xác thực.

Giải quyết vấn đề thực tiễn

Hiện trạng

Các công cụ tìm kiếm truyền thống chủ yếu dựa trên từ khóa, dẫn đến số lượng kết quả lớn nhưng độ nhiễu cao. Người dùng khó nắm bắt được xu hướng tổng thể và những khoảng trống nghiên cứu tiềm năng.

Giải pháp LiemResearch

Áp dụng tìm kiếm ngữ nghĩa, tính toán chỉ số tăng trưởng tự động, sinh báo cáo tổng quan có trích dẫn trực tiếp và phát hiện khoảng trống nghiên cứu dựa trên kho dữ liệu thực.

5. Smart Reasoning Core (F1–F4)

Bốn trụ cột xử lý AI, chia sẻ giữa tất cả tính năng. Mỗi feature gọi đúng trụ cần dùng thay vì tự implement lại.

Smart Reasoning Core

F1 — Shared Retriever

Truy xuất ngữ nghĩa chung

Module retrieval/retriever.ts — dùng chung cho search, report, gap, chat. Nhận query text → embed → $vectorSearch trên Atlas → trả RetrievedPaper[] kèm aiAnalysis nếu có.

4 projection modes: search, report, gap, chat

F2 — Structured Paper Knowledge

aiAnalysis — tri thức trích xuất 1 lần

paper-analysis.worker.ts chạy theo CRON, dùng Gemini trích xuất 8 trường từ title+abstract: summary, methods, dataset, findings, limitations, contributions, futureWork, keyTerms. Lưu vào paper.aiAnalysis, dùng lại cho mọi feature. Fallback: khi chưa có aiAnalysis → dùng abstractText gốc.

F3 — Grounding & Citation

Kiểm định trích dẫn

Module llm/grounding.ts — kiểm tra mọi trích dẫn [n] trong văn bản AI sinh ra có trỏ về bài báo thật trong tập evidence hay không. Loại bỏ nguồn giả mạo/hallucination.

F4 — Cached LLM + Model Routing

Cache & chọn model thông minh

llm.run.ts + llm.factory.ts — cache response trong Redis theo hash(query + paperIds + promptVersion + model). Factory pattern chọn Gemini hoặc Ollama qua LLM_PROVIDER env. Gemini Flash cho tác vụ nhanh, Pro cho deep analysis.

6. Các tính năng AI — Feature Flows

Mỗi tính năng sử dụng các trụ cột F1–F4 ở mức khác nhau. Bảng dưới cho thấy flow thực tế.

Implemented

Semantic Search

Query → F1 Retriever (embed + $vectorSearch) → trả danh sách paper xếp theo cosine score. Không gọi LLM, 0 token. Hỗ trợ sort theo relevance/year/citations, filter paperKind/openAccess/provider/minScore.

Opt-in LLM Re-rank: Khi bật rerank=true, hệ thống gửi top candidate pool cho Gemini Flash chấm lại điểm relevance rồi sắp xếp lại. Nếu LLM fail → negative-cache 10 phút (tránh đốt quota lần sau) rồi graceful fallback về thứ tự vector gốc. Paper bị LLM bỏ sót thì giữ nguyên vector score thay vì bị gán 0.

F1 Retriever · Gemini Embedding · MongoDB Vector · Optional LLM Rerank
Implemented

Trends Dashboard

MongoDB Aggregation Pipeline tính YoY, CAGR, Momentum cho keywords/topics theo năm. Deterministic, không LLM. 3 cổng chống trend giả.

MongoDB Aggregation · Deterministic Compute
Implemented

RAG Reports

Query → F1 retrieve top-K → F4 check cache → Gemini sinh markdown có [n] citations → F3 grounding check → lưu + trả FE. Worker BullMQ xử lý nền.

Deep Analysis fallback: Khi deep analysis mode bị LlmTruncationError (output vượt maxTokens) hoặc LlmContentError (JSON malformed) → hệ thống tự động fallback về classic RAG thay vì fail toàn bộ report. Fan-out gaps vào research_gaps collection sau khi report hoàn tất (non-fatal).

F1 + F3 + F4 · Gemini Flash/Pro · BullMQ report.worker · Auto-fallback
Implemented

Research Gaps

2 luồng: (a) Standalone — user yêu cầu quét chủ đề; (b) Fan-out — tự trích từ report. F1 retrieve → F2 dùng aiAnalysis nếu có → Gemini phân tích gaps kèm confidence + evidence counting trên corpus.

F1 + F2 + F4 · Gemini Flash · gaps.worker
Implemented

Gap Directions

Với mỗi gap, AI gợi ý 2–4 hướng nghiên cứu kèm rationale và relatedPaperIds. On-demand (lazy), không fetch khi mount. Cache force:false ở lần đầu.

F2 + F4 · Gemini Flash · Advisory only
Implemented

Paper Compare

2–4 paper → bảng metrics (year, citation, aiScore) + LLM qualitative comparison (phương pháp, dữ liệu, kết quả, hạn chế). F2 dùng aiAnalysis khi available, fallback abstractText.

F2 + F4 · Gemini Flash · compare.service
Implemented

Project Chat

User hỏi đáp trên tập paper đã lưu trong project. Load papers từ project → dùng embedding vectors của papers để pickEvidence chọn papers gần câu hỏi nhất → LLM trả lời. Provider cắm rút: Gemini mặc định hoặc Ollama local.

Token budget: fitToBudget() kiểm soát prompt size, tự cắt papers và history để fit vào CHAT_MAX_PROMPT_CHARS. Abstract cắt theo CHAT_ABSTRACT_MAX_CHARS. Cache theo hash(projectId + question + paperIds + provider).

F1 + F4 · Gemini hoặc Ollama · projects module · Token budget control
Implemented

MCP / Deep Analysis

Ở chế độ deep analysis, Gemini Pro có thể gọi tool nội bộ (search_papers, get_trends, count_papers) qua MCP executor. AI tự bổ sung evidence trước khi viết report. Tool run logs lưu vào mcp_tool_runs collection.

Graceful error handling: Tool error KHÔNG crash pipeline. Khi tool throw (ví dụ: get_trends trả 404 cho topic hiếm), executor trả { error: message } cho model tự xử lý, thay vì fail cả report qua 5 lần BullMQ retry.

F1 + F3 + F4 · Gemini Pro · mcp.executor + mcp.tools · Graceful tool errors
aiAnalysis fallback: Tất cả AI features (gaps, directions, compare, chat) ưu tiên dùng paper.aiAnalysis (structured knowledge). Khi paper chưa được phân tích bởi worker, hệ thống tự động fallback sang abstractText gốc — đảm bảo mọi paper đều có thể tham gia AI pipeline.

7. Local LLM Provider - Ollama

Vai trò của Ollama trong hệ thống

Ollama là gì?

Ollama là một runtime chạy LLM local trên máy backend, tương thích OpenAI API format. Hệ thống implement OllamaLlmProvider (tuân thủ LlmProvider interface) để cắm vào cùng pipeline với Gemini.

Khi LLM_PROVIDER=ollama trong .env, toàn bộ lệnh generate() chuyển sang Ollama thay vì gọi Gemini API — giảm quota generate của Gemini cho chat/demo.

Giới hạn quan trọng

Khía cạnhOllamaGemini
Text Generate✅ Thay thế được✅ Mặc định
Embedding❌ Không thay thế✅ Luôn dùng Gemini
MCP / Function Calling❌ Không hỗ trợ✅ Gemini Pro
Chất lượng outputPhụ thuộc model localCao, ổn định
Lưu ý: Ollama chỉ thay thế phần generate text (chat, một số report đơn giản). Embedding vẫn luôn dùng Gemini vì vector index đã build trên Gemini 768d — chuyển embedding model sẽ phá index.

11. Cơ chế chấm điểm bài báo (v2 — công bằng theo tuổi)

Điểm bài báo là điểm nội tại, tính bằng công thức cố định — tái lập được, không phải số do AI sinh ra.

finalScore = 0.60 × Impact + 0.40 × Recency
Tiêu chíTrọng sốCông thứcÝ nghĩa
Impact60%log10(citations/tuổi + 1) / log10(101)Ảnh hưởng theo tuổi — bài 2024 không bị thiệt vì chưa kịp tích citation
Recency40%1 − (năm nay − năm XB)/10Độ mới — bài > 10 năm = 0

Độ đầy đủ dữ liệu (7 trường) hiển thị RIÊNG, không trộn vào điểm giá trị.

Hệ thống chấm chất lượng 6 chiều (paper-quality.ts)

Ngoài điểm giá trị (impact + recency), mỗi paper còn được chấm qualityScore — quyết định tier, chi phí download và phần thưởng upload.

Chiều chấmĐiểm tối đaTiêu chí
Metadata15Có title (3), authors (3), year (2), abstract (4), keywords/topics (3)
Source15DOI hợp lệ (15), link arxiv/zenodo (8), link khác (10), chỉ PDF (2)
Duplicate20Unique (20), needs review (10), duplicate (0)
Relevance15Có topics + keywords + abstract ≥ 20 words → 15; 2/3 → 10; 1/3 → 5
Prestige15article/proceedings (15), review/book-chapter (10), preprint (8), khác (3)
Utility15data/software/benchmark (15), method/experiment/result (12), review (12), case study (8)

qualityScore = Σ 6 chiều (0–95) → map vào 5 tiers:

TierTênĐiểmDownload CostUpload Reward
0Không hợp lệ0–49N/A0 credit
1Cơ Bản50–6420 credit30 credit
2Chuẩn Học Thuật65–7930 credit60 credit
3Giá Trị Cao80–9150 credit100 credit
4Tinh Hoa92–10080 credit150 credit
Quan trọng: Upload reward được thiết kế sao cho bài tier cao nhất chỉ fund ~1.5 lần request (100 credit/request), tránh lạm phát credit. Papers đã nhận reward (uploadRewardedAt set) sẽ freeze reward cũ, không bị ảnh hưởng khi điều chỉnh bảng tier.

Bổ sung: AI chấm bài on-demand

Điểm công thức là tất định. Riêng với bài người dùng upload, hệ thống cho gọi Gemini chấm theo yêu cầu — 3 chiều (1–5): relevance, groundedness, completeness. Thuần tư vấn: KHÔNG đổi tier, KHÔNG đổi credit, KHÔNG tự duyệt. Cache theo từng bài, rate-limit 20 lượt/giờ/user.

12. Làm sao biết một xu hướng là THẬT?

Chủ đề chỉ được xem là "đang lên" khi vượt qua 3 cổng chống trend giả.

Cổng lọcChặn cái gì
Base ≥ 2 bài năm trước (MIN_RISING_PREV_BASE)Chặn 1 → 9 bài = "+800%" giả, vô nghĩa
Bỏ năm hiện tại đang chạy dở (truncateToCompleteYears)Năm chưa hết → chưa đủ dữ liệu → không lấy làm chuẩn
3 chỉ số độc lập (YoY + CAGR + Momentum)Trend thật phải đẹp ở cả 3, không chỉ spike 1 năm

Công thức Gap Evidence Scoring (gap-evidence.ts)

Khi AI đề xuất một gap ("chủ đề A giao B còn ít nghiên cứu"), hệ thống tự đếm trên corpus để kiểm chứng:

BướcCông thức / Logic
1. Đếm giao điểmintersectionCount = số paper chứa CẢ topicA VÀ topicB (regex trên title+abstract)
2. Đếm nhánh chaparentCounts.a, parentCounts.b = số paper mỗi nhánh riêng
3. Tính ceilingceiling = max(scarceAbs, floor(scarcePct × minParent), 1)
4. Scarcity scorescarcityScore = clamp01(1 - intersectionCount / ceiling) — càng cao càng hiếm
5. Parent risingLấy YoY growth % của nhánh cha tăng nhanh hơn, so với ngưỡng parentRisingMin
6. Confirmedconfirmed = scarce AND parent rising — gap thật = hiếm VÀ nhánh cha đang tăng
7. Evidence confidenceConfirmed: 0.5 + 0.5 × scarcityScore. Unconfirmed: 0.25 × scarcityScore
Ý nghĩa: Confidence là số thật từ dữ liệu, không phải AI tự khai. Gap "confirmed" (hiếm + nhánh cha đang tăng) được anchor ở 0.5–1.0, gap unconfirmed bị giữ ở 0–0.25. FE sort theo evidenceConfidence giảm dần.

16. Cơ chế chịu lỗi (Error Resilience)

Hệ thống thiết kế để không đốt quota vô ích và degrade gracefully thay vì crash.

Cơ chếMô tảÁp dụng ở
LlmTruncationError Output vượt maxOutputTokens → đánh dấu nonRetryable = true. BullMQ worker không retry (retry cùng budget sẽ fail y hệt). Fail fast thay vì đốt 5× quota. Report worker, Gap worker
LlmContentError JSON malformed, report quá ngắn, citation out-of-range → nonRetryable. Cùng input sẽ sinh cùng lỗi, retry vô ích. Report worker, Gap worker, Compare
Deep Analysis auto-fallback Khi deep analysis (Gemini Pro + tools) bị Truncation/Content error → tự động fallback về classic RAG (generateJSON không tools). Report vẫn hoàn thành. rag.service.ts
Rerank negative-cache Khi LLM rerank fail → cache failure 10 phút. Cùng query+filters trong 10 phút sẽ skip rerank, trả kết quả vector gốc — tránh đốt quota lặp lại. search.service.ts
MCP tool error isolation Tool throw → trả { error: message } cho model. Model có thể proceed hoặc fallback. Tool error KHÔNG fail pipeline. mcp.executor.ts
Non-fatal side effects Audit logging, gap fan-out, MCP tool-run logging — tất cả .catch() non-fatal. Nếu ghi log fail, report/gap vẫn thành công. rag.service, gaps.service, mcp.executor
